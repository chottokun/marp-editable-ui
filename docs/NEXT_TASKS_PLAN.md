# 次期機能実装計画書: Marp Editable UI 強化案

本ドキュメントは、Marp Editable UI の今後の機能拡張に関する技術的検討と実装計画をまとめたものです。

---

## 1. スライドコンテンツの溢れ防止（自動レイアウト調整）

### 課題
テキスト量や画像サイズが1枚のスライド許容範囲を超えた場合、下端が突き抜けて見えなくなる（オーバーフロー）。

### 解決アプローチ
*   **Marp Core Fitting 機能の強制適用**:
    *   `<style scoped> section { font-size: auto; }</style>` をバックエンドで自動注入し、Marp Core の自動フォントサイズ調整機能を全スライドで有効化する。
*   **AI によるページ分割の最適化**:
    *   システムプロンプトにおいて「スライド1枚あたりの文字数制限」を明示し、超過が見込まれる場合は論理的な区切りで `---` を挿入するよう AI を再学習させる。
*   **レンダリング後の高さ検知と自動分割**:
    *   （高度な実装）ヘッドレスブラウザでレンダリング結果の要素高さを取得し、スライド高（720px等）を超える場合に、適切な位置で Markdown を物理的に分割する。

---

## 2. プレゼンテーションモードの実装

### 課題
編集画面ではなく、プロジェクター投影や共有に適した全画面表示での発表機能。

### 解決アプローチ
*   **Fullscreen API の統合**:
    *   ブラウザ標準の `requestFullscreen()` を使用。プレビューコンポーネントのみを全画面化し、UIノイズ（エディタ、ヘッダー）を排除する。
*   **キーボード/クリック制御**:
    *   左右矢印キー、スペースキー、マウスクリックによるページ送り・戻しを実装。
*   **専用ルート `/present` の新設**:
    *   React Router を利用し、現在の Markdown データをステート（または URL クエリ）として引き継ぐ「発表専用ページ」を作成。
*   **ページ番号同期**:
    *   エディタで選択中のスライドとプレゼンモードの表示ページを双方向に同期させる。

---

## 3. メイン画面でのスライド可変スケーリング

### 課題
分割画面（プレビューエリア）のサイズを変更した際、スライドのサイズが固定（あるいはスケールが不適切）で、視認性が低下する。

### 解決アプローチ
*   **ResizeObserver による動的スケーリング**:
    *   プレビューエリアの親要素サイズをリアルタイム監視。
    *   CSS `transform: scale()` を使用し、アスペクト比を維持したまま、親要素の幅と高さの両方にフィットする最大倍率を自動計算。
*   **CSS Container Queries の活用**:
    *   プレビュー容器に対して Container Queries を適用し、スケールに合わせた細かなスタイル微調整（余白やフォントの微調整）を行う。
*   **実装計算式（例）**:
    *   `scale = Math.min(containerWidth / 960, containerHeight / 720)`

---

## 実装優先順位の検討（案）

1.  **スライドの可変スケーリング**: 最も利用頻度が高く、編集体験を劇的に改善するため。
2.  **プレゼンテーションモード**: ツールとしての完成度（「作る」から「見せる」まで）を完結させるため。
3.  **自動レイアウト調整**: AIのプロンプト改善で一定の効果が見込めるため、システム的な自動分割は最終ステップとして検討。
